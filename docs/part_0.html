Part 0: Understanding Design Principles

Introduction

Designing software isn't just about deciding what code to write. It's about understanding how different pieces of the system will interact, predict potential issues, and ensuring that the system can evolve without needing a complete rewrite. Before you dive into coding, understanding these principles can make the difference between software that lasts and software that needs constant fixing.

Why Are Designs Important?

  • Clarity of Thought: Designing before coding helps crystallize our ideas. It allows us to identify and fix conceptual mistakes before they turn into code bugs. For example, by sketching out the data flow for a URL shortener, you can identify potential bottlenecks or security issues early on.

  • Efficient Collaboration: A clear design serves as documentation. It provides a roadmap for teams, or even just yourself, ensuring that everyone understands the bigger picture and their role in it. This is especially important in a team setting where different members work on various parts of the project.

  • Scalability: Designing with scalability in mind ensures that the system can handle growth, be it more users, more data, or both. Consider how your URL shortener might need to scale if it becomes popular and handles millions of requests per day.

  • Fault Tolerance: Good designs consider failures. How will the system recover? What are the potential points of failure? For instance, think about what happens if your database goes down or if there's a spike in traffic.

  • Maintainability: A well-designed system is modular and easy to modify. As business requirements change and projects evolve, your software can adapt without extensive rewrites. This could mean separating your URL shortening logic from your user management logic.

  • Cost-Efficiency: A well-thought-out design can reduce development time, reduce the need for refactoring, and lead to more efficient resource use in the cloud or on-premises. Optimizing your design can save resources and reduce operational costs.

Design-First Approach

  • Thinking Before Coding: While it's tempting to jump straight into coding, taking a step back to design can save a lot of headaches later on. This involves creating design documents, diagrams, and detailed plans before writing any code.

  • Iterative Process: Design isn't a one-off step. As requirements evolve and you understand the problem better, revisit and refine your designs. Continuous iteration and feedback are key to improving your design over time.

  • Picturing What You Do: Visualize how components will interact. It can be simple sketches or more detailed digital wireframes. Tools like Excalidraw can be very helpful for this.

Design Requirements

Your design doc should address the following concerns. It will be continually be updated as requirements get added and/or changed. You can get the template here. Please always keep your coach in the loop as only they (much like your stakeholders and team lead) can approve your design.

  1. Requirements. All designs must have requirements based on the use cases described by the problem statement. In the industry, problem statements are usually vague, you will need to learn to infer what the requirements are to be based on descriptions provided by your team lead (TL) and stakeholders.

  2. Overall Architecture. The design doc should cover all the components the system or service should have. Furthermore, given the functional requirement, decide how the data will flow from user (front-end) to (service) back-end to database, and vice versa.

  3. API Design. APIs, much like your functions, usually expects an input and returns an output. APIs require specifications prior to development and implementations primarily on what they do and how they work. In the Design Doc,

  4. You will first need to decide how many APIs will be need to the user.

  5. Then, you need to specify what input your APIs will be expecting from the user and what the APIs will return. This will have to be done for all APIs. We have listed what some of them should be down below. If there are any extras please fill them out.

  6. Finally, you will need to detail how each of them will work. Where fill they fetch the data? How would you generate a short URL? etc.

  7. Database Schema and Data Model Design. What are the key data models you will use? You should outline the nature of the data models and for each one their primary keys, and how they interact with other tables if needed. This should include the data types of each field in your data models.

  8. While crafting your design, emphasize visualizing how the service operates. As we're emphasizing a design-first approach, think of where certain components or sections should be placed in the overall structure. This might involve diagrams or flowcharts that showcase how components interact using specific protocols. Whether or not to include drawings is usually up to you, but a well-visualized design can aid in better understanding, so we strongly recommend it.

  9. Code Structure. Instead of having all your code in one file, break it down into separate modules based on functionality. For example, all database-related functions can be in a database.py file, while API route handlers can be in an api_handlers.py file. This ensures better code maintainability and readability. You'll need multiple models and directories. Below, are the directories you will use. In your design, you need to write which files you will use and which directory they go in. One hint is that the main.py​ which starts the application will go in the root directory / . Directories you will use:

  10. /models​ - This includes your database models.

  11. /api​ - Will hold all the API handlers related to your code.

  12. /service​ - Holds the core logic of your application. For example, the code here will connect to read/write from the database as well as shorten the URLs

  13. Database Testing or Mocking. DynamoDB does not provide a local database to test on. Therefore when you run unit tests, you cannot use a local database out of the box. Here are some options. You could mock the dynamo database using Python's unittest. However, you also could structure your code in a way that only the core functions should be tested.

  14. Testing Plan. You will eventually deploy this application on AWS. How will you test it? You should include unit tests and end to end tests. What are common edge cases? What are tricky input/output scenarios should you be wary of?

Exit Criteria

Every project or system has a goal, and it's crucial to define success from the outset.

  • Complete your Design Document: Define what must be achieved by the end of the project. What are the non-negotiable features? What does the system do? How will you test your system? Answer this and many other questions as you design, then get this approved by your Coach.

Conclusion

Understanding the design principles is the foundation of any software development process. As you move through this project, continually refer back to these principles. They'll guide your decisions and ensure that you're not just thinking about code, but crafting a solution.
